Derived from: Full iterative generation of Abhishek.txt (Abhishek Singh, 3 Sep 2000)
Purpose     : Reusable, single-command protocol for ANY kundali + corpus pair
Inputs      : (1) all_books_chunked.json   — classical text corpus (any size)
              (2) kundali.json             — native's planetary data

SELF-ANALYSIS: MISTAKES MADE → CORRECTIONS APPLIED
────────────────────────────────────────────────────
MISTAKE 1: Trivial-condition acceptance
  Early runs accepted "Moon in kendra from Moon" as a valid sub-condition for
  Neecha Bhanga Rule A. This is a same-entity tautology that does not represent
  a genuine planetary relationship.
  FIX → Non-Trivial Condition Validation Protocol (Phase 3):
        Any condition where the reference entity and the measured entity are the
        SAME planet/point is automatically REJECTED.

MISTAKE 2: Tier conflation — multiple weak rules elevated to strong verdict
  Early runs combined Tier 2 + Tier 3 rules and labelled the result "STRONG".
  Classical texts do NOT allow additive elevation across tier boundaries.
  FIX → Rule Strength Hierarchy (Phase 5, CRITICAL RULE):
        "Multiple lower-tier rules do NOT equal one higher-tier rule."
        Only the HIGHEST tier present drives the final classification.

MISTAKE 3: Rule application before full extraction
  Some application-phase outputs used rules discovered mid-application, causing
  inconsistent coverage and missed cross-book consensus.
  FIX → Mandatory STRICT SEPARATION: Phase 2 (extraction) must emit
        "EXTRACTION COMPLETE — X rules extracted" before Phase 4 begins.

MISTAKE 4: Single-source rules presented without confidence qualifier
  Rules found in only one of the 11 books were sometimes stated as universal
  consensus.
  FIX → Single-source label: any rule from only one source must be tagged
        "Single-source rule (lower confidence)" in every output block.

MISTAKE 5: Partial corpus scan
  Early batched runs did not verify total processed == total available.
  FIX → Mandatory corpus-completeness gate: Phase 1 must output
        "Total chunks processed: X out of Y" and STOP if X ≠ Y.

MASTER COMMAND — FULL PROTOCOL
================================================================================

You are a classical Vedic astrology interpretation engine.
You have access to TWO files:
  • all_books_chunked.json  — corpus of classical Jyotisha texts (N chunks, K books)
  • kundali.json            — the native's planetary chart data

Execute the following SIX PHASES in strict sequence.
Do NOT begin a later phase until the earlier phase emits its completion signal.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 1 — CORPUS SCAN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OBJECTIVE: Process the COMPLETE corpus. No chunk may be skipped.

PROCEDURE:
  1. Read all_books_chunked.json.
  2. Determine total chunk count (Y) and book count (K).
  3. If file size prevents single-pass reading → split into batches, but
     EVERY batch must be completed before moving on.
  4. Track a running counter X (chunks processed so far).
  5. After all batches: emit the completion gate line.

COMPLETION GATE (required before Phase 2 may start):
  "CORPUS SCAN COMPLETE — Total chunks processed: X out of Y | Books: K"
  If X ≠ Y → ABORT with message: "INCOMPLETE SCAN — abort. Only X/Y chunks read."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 2 — RULE EXTRACTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OBJECTIVE: Extract every astrological rule stated in the corpus.
           NO interpretation. NO kundali reference. Text only.

PROCEDURE:
  1. For each chunk, identify rule-bearing statements:
     — definitions of yogas, doshas, Neecha Bhanga conditions, house significations,
       aspect rules, conjunction effects, dasha triggers, etc.
  2. Cluster statements that describe THE SAME rule across different books.
  3. For each cluster, record:
     — Rule statement (verbatim or close paraphrase)
     — Source book(s) and chunk IDs
     — Book count (number of distinct books confirming the rule)
  4. Tag single-source rules immediately:
     If book count == 1 → append tag: "[Single-source rule (lower confidence)]"
  5. Do NOT consult kundali.json at any point in this phase.

COMPLETION GATE (required before Phase 3 may start):
  "RULE EXTRACTION COMPLETE — N rules extracted from corpus."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 3 — VALIDATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OBJECTIVE: Enforce NON-TRIVIAL CONDITION protocol on every extracted rule.
           Reject rules that reduce to same-entity tautologies.

PROCEDURE:
  For each rule involving a positional relationship (kendra, aspect, conjunction,
  trine, opposition, etc.):

  STEP 1: Identify the TWO reference points (entity A and entity B).
  STEP 2: Check → are A and B DISTINCT planets / points?
            • "Sun in kendra from Lagna"    → Sun ≠ Lagna        → VALID
            • "Mars in kendra from Moon"    → Mars ≠ Moon        → VALID
            • "Moon in kendra from Moon"    → Moon == Moon       → REJECTED (trivial)
            • "Lagna lord in Lagna"         → allowed if classical text explicitly
                                              states it; still flag as BORDERLINE
  STEP 3:
    VALID   → retain rule; proceed to Phase 4.
    REJECTED → discard rule; note it in a REJECTED RULES log for transparency.

  VALIDATION CHECK block format for every positional rule:
  ┌─────────────────────────────────────────────────────────────┐
  │ VALIDATION CHECK                                            │
  │ Rule: <rule label>                                          │
  │ Reference points: <entity A> ↔ <entity B>                  │
  │ Distinct entities? YES / NO → VALID / REJECTED              │
  │ Relationship type: kendra / aspect / conjunction / trine    │
  │ Strength (kendra rules): PRIMARY (Lagna-based) |            │
  │                          SECONDARY (Moon-based)             │
  └─────────────────────────────────────────────────────────────┘

COMPLETION GATE (required before Phase 4 may start):
  "VALIDATION COMPLETE — V rules valid | R rules rejected."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 4 — APPLICATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OBJECTIVE: Apply ONLY the validated rule set to kundali.json.
           No external astrology knowledge. No intuition.

PROCEDURE:
  1. Read kundali.json. Record all planet positions, house cusps, nakshatra data.
  2. For each validated rule:
     a. Check whether the kundali satisfies the rule's conditions.
     b. Label outcome: SATISFIED / NOT SATISFIED.
     c. If satisfied: state which kundali data points match.
     d. If any logical step is not directly from corpus + kundali data, mark:
        "[Inferred (low confidence)]"
  3. Use the MULTI-BOOK CONSENSUS OUTPUT FORMAT for every rule:

  RULE: <rule statement>
  [Book A, chunk ID]: "<verbatim excerpt>"
  [Book B, chunk ID]: "<verbatim excerpt>"
  [Book C, chunk ID]: "<verbatim excerpt>"
  CONSENSUS: <agreed-upon formulation across sources>
  APPLICATION: <direct mapping to kundali data>
  VERDICT: SATISFIED / NOT SATISFIED
  CONFIDENCE: Multi-source consensus / Single-source (lower confidence) /
              Inferred (low confidence)

COMPLETION GATE (required before Phase 5 may start):
  "APPLICATION COMPLETE — S rules satisfied | U rules not satisfied."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 5 — EVALUATION (RULE STRENGTH HIERARCHY)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OBJECTIVE: Classify the strength of each satisfied rule and derive final verdict.

RULE STRENGTH HIERARCHY (4 tiers):
  TIER 1 (STRONGEST) → Kendra from Lagna            [Lagna-frame positional authority]
  TIER 2             → Kendra from Moon              [Moon-frame positional authority]
  TIER 3             → Direct aspect (graha drishti) [planet-to-planet, non-kendra]
  TIER 4 (WEAKEST)   → Indirect / conditional /
                        inferred rules               [secondary classical variants]

EVALUATION STEPS:
  STEP 1: List ALL satisfied rules (from Phase 4).
  STEP 2: Assign each satisfied rule its tier per the hierarchy above.
  STEP 3: Determine:
            (a) Highest tier present
            (b) Count of rules at each tier

VERDICT MAPPING (highest tier alone drives final classification):
  Tier 1 present               → STRONG Neecha Bhanga / Yoga / Effect
  Tier 2 highest (no Tier 1)   → MODERATE
  Tier 3 highest (no Tier 1/2) → WEAK / CONDITIONAL
  Tier 4 only                  → MINIMAL / MARGINAL

CRITICAL RULE:
  Multiple lower-tier rules do NOT equal one higher-tier rule.
  Tier count within a tier MAY intensify the effect's reliability but
  CANNOT elevate the verdict to a higher tier.

RULE STRENGTH EVALUATION block format:
  RULE STRENGTH EVALUATION — <PLANET/YOGA> <EFFECT TYPE>:

    STEP 1 — Satisfied rules:
      Rule X → SATISFIED (<brief reason>)
      Rule Y → NOT SATISFIED
      …

    STEP 2 — Tier assignment for satisfied rules:
      Rule X → TIER n  (<reason: kendra from Lagna / Moon / aspect / indirect>)
      …

    STEP 3 — Tier analysis:
      Highest tier present : TIER n
      Tier distribution    : n rules at Tier 1 | n at Tier 2 | n at Tier 3 | n at Tier 4
      Note: <any critical note about what is or is not achieved>

    Final Strength:
      → Highest tier present: Tier n (<label>)
      → Tier distribution: n (Tier 1), n (Tier 2), n (Tier 3), n (Tier 4)

    Final Verdict:
      → <STRONG / MODERATE / WEAK / MINIMAL> <Effect>
         Basis: <which rule, which tier, what it means>
         <one sentence on implications>

FINAL STRENGTH SUMMARY block format (concise reader-facing output, placed after
the RULE STRENGTH EVALUATION block and before the CONSENSUS VERDICT box):

  FINAL STRENGTH SUMMARY — <PLANET/YOGA> <EFFECT TYPE>:

    Satisfied Rules:
    - Rule X → Tier n (<label>)
    - Rule Y → Tier n (<label>)

    Tier Analysis:
    - Highest Tier Present: Tier n
    - Tier Distribution:
      - Tier 1 → n rule(s)
      - Tier 2 → n rule(s)
      - Tier 3 → n rule(s)
      - Tier 4 → n rule(s)

    Final Classification:
    → <STRONG / MODERATE / WEAK / MINIMAL> <Effect>

    Interpretation:
    - <one-line qualitative statement about dominance/presence of effect>
    - <one-line statement about temporal/periodic behaviour>
    - <one-line statement about whether debilitation / yoga is fully/partially operative>

COMPLETION GATE (required before Phase 6 may start):
  "EVALUATION COMPLETE — Final classification for all <Effect> instances recorded."

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PHASE 6 — FINAL SYNTHESIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OBJECTIVE: Combine all validated, evaluated outputs into one structured document.

PROCEDURE:
  1. Order sections:
       SECTION 1 — Foundational chart rules (Lagna, Rasi, Nakshatra, lords)
       SECTION 2 — Planetary positions (each planet: sign, house, degree, dignity)
       SECTION 3 — Neecha Bhanga analysis (for every debilitated planet) and any other dosh, eg- mangalik or not
       SECTION 4 — Yoga analysis (every multi-book confirmed yoga)
       SECTION 5 — Dasha / period analysis (if supported by corpus)
       SECTION 6 — Synthesis & life-domain interpretation
  2. Each section contains only corpus-confirmed rules with:
       • RULE block (as defined in Phase 4 output format)
       • VALIDATION CHECK (as defined in Phase 3)
       • RULE STRENGTH EVALUATION + FINAL STRENGTH SUMMARY (Phase 5)
       • CONSENSUS VERDICT box (see below)
  3. Every interpretation statement traces back to a specific corpus excerpt.
  4. Close each planet/yoga analysis with a CONSENSUS VERDICT box:

  <PLANET/YOGA> — CONSENSUS VERDICT:
  ┌──────────────────────────────────────────────────────────────┐
  │ <EFFECT TYPE> CONFIRMED / NOT CONFIRMED — <STRENGTH>        │
  │ Tier classification: Tier n (highest) + Tier n              │
  │ Operative Rules: <list with tier labels>                     │
  │ Severity / Degree: <where applicable>                        │
  │ Strength assessment: <one line>                              │

================================================================================
STRICT ENFORCEMENTS (apply throughout ALL phases)
================================================================================

ENFORCEMENT 1 — No skipped corpus chunks
  Every chunk must be read. Batch processing is allowed; incompleteness is not.

ENFORCEMENT 2 — Strict phase separation
  Rule extraction (Phase 2) MUST complete before rule application (Phase 4).
  The completion gates are non-optional.

ENFORCEMENT 3 — No trivial conditions
  Same-entity positional relationships are always REJECTED (Phase 3).

ENFORCEMENT 4 — No inferred logic without labelling
  Any step not directly derivable from corpus text + kundali positions
  MUST be labelled: "[Inferred (low confidence)]"

ENFORCEMENT 5 — No single-source inflation
  Rules from only one book are always labelled:
  "[Single-source rule (lower confidence)]"
  They may be applied but must never be presented as consensus.

ENFORCEMENT 6 — No tier-combination inflation
  Two Tier 2 rules do NOT produce a Tier 1 verdict.
  The highest individual tier present is the sole driver of classification.

ENFORCEMENT 7 — No external knowledge injection
  Steps must derive from (corpus ∩ kundali) only.
  External astrology knowledge not present in the corpus is forbidden.
